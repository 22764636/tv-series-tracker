SeriesDetail page layout

Two tiers. The header (poster + a column beside it) holds only what's compact: title (with "Aggiorna da TMDB"/"Elimina serie" as right-aligned icon buttons sharing the title's own row — see below), status pills, progress bar + "X/Y episodi visti" (+ "Z rimanenti" when known), link, Wikipedia EN/IT. Everything heavier — giorni di visione (7 pills) and valutazione + grafico — is instead its own full-width section below the header, each wrapped in rounded-2xl border border-border bg-surface p-4, stacked with flex flex-col gap-4. Don't fold those two back into the header column: that's what made the page read as cramped and inconsistent with the full-width season/episode section further down. Below sm, the header itself stacks vertically (flex-col items-center → sm:flex-row sm:items-start): poster centered on top, info column full width underneath — always-side-by-side broke down at larger accessibility text sizes, wrapping every row inside an already-narrow column into ragged multi-line spacing.

Rating — two hearts (SeriesDetail + card)

RatingRow in SeriesDetail.jsx — only rendered when series.status === 'completed' (not for renewed). Two independent HeartRating rows, one per person: 💙 (series.ratingBlue) and 💜 (series.ratingPurple) — these two emoji are an explicit, one-off exception to the no-emoji rule, requested for this exact spot only. Each follows the same edit/display/add pattern as LinkRow: plain text "💙 X/10", a small text-xs text-muted "Modifica" button, or a "💙 + Aggiungi" text-accent hover:underline link when unset. Input: type="number" min="1" max="10" step="0.01" (up to 2 decimals), same input classes as the link field. Above the two heart rows, when at least one is set, a plain "Valutazione: X/10" line shows the averageRating() (progress.js) of whichever hearts have voted — if only one has, that single value stands in as the provisional average. formatRating() rounds to 2 decimals and strips trailing zeros (8 instead of 8.00). On the card, shown as "Voto X/10" using the same average — the status-badge row uses flex items-center justify-between so the rating sits flush right opposite the badge, instead of just trailing it. The "Modifica"/"+ Aggiungi" control on these two total rows disappears entirely (HeartRating's locked prop, only ever passed here — never for the per-episode rows in SeasonBlock) once every episode already has that heart rated: see heartFullyRated() in progress.js for why it's "every episode", not "any episode".

Icon family (Search / Refresh / Close / Share)

SearchIcon.jsx, RefreshIcon.jsx, CloseIcon.jsx and ShareIcon.jsx are deliberately one visual family — viewBox="0 0 24 24", stroke="currentColor", stroke-width="2", stroke-linecap="round" — instead of mixing hand-drawn SVGs with Unicode text glyphs (↻, ✕) as earlier versions did: those glyphs render inconsistently across device fonts (often visibly smaller/thinner than other icons at matched font-size), and even after fixing the size, a font glyph and a hand-drawn SVG still don't look like the same design language. CloseIcon.jsx replaces every ✕ in the app (Modal close button, SeriesDetail delete button, search clear buttons, remove season row in AddSeriesModal) — one component, one look, wherever an "X" close/remove/clear action appears. RefreshIcon.jsx is the classic two-arrow circular refresh (refresh-cw), not a single 270° arc with one arrowhead tucked in a corner — more symmetric and reads as "refresh" faster at a glance. ShareIcon.jsx is the three-node share glyph.

Title-row icon buttons (Condividi / Aggiorna da TMDB / Elimina serie)

Right-aligned in the same flex row as the <h1> title (flex items-start justify-between), in this order (least to most destructive): Condividi, Aggiorna (TMDB series only), Elimina. All three flex h-8 w-8 items-center justify-center rounded-full text-muted hover:bg-surface-hover active:bg-surface-hover — a fixed box rather than content-driven padding, so every button is the same size regardless of icon. Share/refresh hover/active to text-accent, delete to text-danger; all three get the same active: treatment as hover: since hover never fires on touch. Refresh (series.source === 'tmdb' only) spins (animate-spin) while pending; an error renders as a text-xs text-danger line just under the title row, no modal/toast. Share calls navigator.share when available, else falls back to navigator.clipboard.writeText with a confirmation line in that same spot ("Link copiato negli appunti" / an error message if the clipboard write itself fails). Delete opens ConfirmDeleteModal (below) instead of acting immediately.

Wikipedia links (SeriesDetail)

WikipediaRow/WikipediaLink — "Wikipedia (EN)" / "Wikipedia (IT)" links, defaulting to the computed guess (wikipediaUrl, wikipedia.js) but overridable per series — same edit/display pattern as LinkRow (plain link + small "Modifica" text button, inline form with Salva/Annulla when editing). Clearing an override falls back to the computed guess rather than removing the link entirely.

Confirm-delete modal

ConfirmDeleteModal reuses Modal — never a native window.confirm(), which looks out of place and is too easy to accept by reflex compared to a styled dialog with the destructive action visually distinct. Body: one line of text-sm text-muted explanatory text naming the series being deleted. Footer: flex justify-end gap-2 — "Annulla" (rounded-full border border-border px-4 py-2 text-sm font-medium text-text hover:bg-surface-hover) then "Elimina" (rounded-full bg-danger px-4 py-2 text-sm font-medium text-white hover:opacity-90, the one place a solid danger-filled button is used, deliberately not danger-soft, since this is the single confirm step for an irreversible action). The same exact pattern (body copy + same two footer buttons) is reused unchanged in SeasonBlock for confirming "Segna non vista" on a whole season — same visual weight because it's an equally real loss of data (watched dates), not a lighter "are you sure" aimed only at series deletion.

Search (desktop header + mobile FAB)

Same filter logic everywhere (filterSeriesByTitle, src/lib/search.js — plain case-insensitive substring on the title, capped to 8 results), two different surfaces per platform:

Desktop (HeaderSearch.jsx, hidden sm:block): inline rounded-full border border-border bg-surface input in the header, widening slightly on focus. A dropdown (absolute ... rounded-xl border border-border bg-surface shadow-xl) lists matches as poster-thumbnail + title rows, same shape as AddSeriesModal's search results. Always enabled, including on a series' own detail page (jumping to another series from there is a legitimate thing to want). Keyboard shortcut / focuses it (ignored while typing in any other field); Escape clears and blurs, Enter jumps to the first result.

Mobile (MobileSearchModal.jsx + a floating button in the global App.jsx shell — not a page component, so it stays visible on every route, not just Libreria — sm:hidden): a fixed bottom-5 right-5 circular bg-accent-solid button, carrying SearchIcon.jsx (a small hand-drawn SVG magnifier — not emoji, see above). Tapping it opens a Modal with the same search input + result list pattern as the desktop dropdown, just full-width inside the modal instead of an anchored popover.

Both variants get a clear (CloseIcon.jsx) button inside the input, shown only once there's text, positioned absolute right-2(.5) top-1/2 -translate-y-1/2.

Per-episode rating + duration rows (SeasonBlock)

Below each season's episode-button grid, one row per relevant episode: flex items-center justify-between gap-2 rounded-xl border border-border bg-surface px-3 py-2 text-sm, same row shell as the Calendario entry list. Episode label (text-muted) on the left; on the right, duration then rating. A watched episode always gets a row (for its rating); a TMDB episode also gets one once TMDB has published a duration for it even if unwatched; a manual-source episode always gets one (duration must be enterable before it's watched, so remaining-time has something to sum). Duration: plain text-xs text-muted text for TMDB (automatic, not editable — it would just get overwritten on the next "Aggiorna da TMDB" anyway), or the same edit/display/add pattern as LinkRow for manual series (DurationField). Rating: the same two HeartRating widgets (💙/💜) as the total rating, shown only once the episode is watched — wired to setEpisodeRating instead of setRating, available regardless of the series' own status (unlike the total rating, still gated to "Completata").

RatingChart (SeriesDetail)

src/components/RatingChart.jsx — dependency-free responsive SVG line chart, shown under the total rating whenever at least one episode has a rating. One point per rated episode (x-axis label S1E1, S1E2, ... skipping labels that would collide rather than ever overlapping them), three 2px lines: 💙 and 💜 in var(--chart-blue)/var(--chart-purple), plus a dashed var(--color-muted) "Media" line (the average is a derived value, not a third voter, so it deliberately isn't a third categorical hue). Above the chart, a row combines the legend swatches with the current totals — "💙 X/10", "💜 X/10", "Media X/10" — always visible, not just on hover; the caller passes those numbers in as props (totalBlue/totalPurple/ totalAverage, computed with progress.js helpers) rather than the chart re-deriving them. Crosshair + one shared tooltip on pointer move/hover, listing every series' value at that episode — every value it shows is also plainly visible in the per-episode rows above (that's this chart's "table view"). Sizing: SVG viewBox defines the logical coordinate space, and the element is pinned to that exact pixel size (style={{ width, height: HEIGHT }}, never width: 100%) — 1 SVG unit always equals 1px. A percentage width was tried first and was a real bug: it stretches the whole coordinate system to fill the card, so a short chart (few rated episodes) on a wide desktop card rendered comically oversized labels/dots/lines — "standard style regardless of rated count" means extra container width is just left empty next to a narrower chart, not used to inflate it. Falls back to horizontal scroll (overflow-x-auto wrapper, plain scrollbar, not .no-scrollbar) once episodes (or a high zoom level, see below) no longer fit the container.

Touch — onTouchStart/onTouchMove alongside the pointer handlers, reading e.touches/e.changedTouches directly; relying on onPointerDown/onPointerMove alone misses real touch taps in practice. onPointerLeave ignores pointerType === 'touch' — lifting a finger fires a pointerleave too, which would otherwise instantly clear the tooltip a tap just set; only an actual mouse leaving the chart area should dismiss it. No touch-action restriction on the SVG (a previous version set pan-y specifically to keep vertical page scroll working through the chart) — real bug: pan-y also told the browser to treat horizontal swipes as script-only, which silently broke the ancestor overflow-x-auto's native horizontal scroll, leaving longer seasons stuck showing only what fit in the initial viewport. None of these handlers call preventDefault, so the default (auto, native on both axes) already scrolls each direction correctly on its own nearest scrollable ancestor.

Zoom — two small round h-6 w-6 rounded-full border border-border "−"/"+" buttons (plain ASCII, not an SVG icon — unambiguous and renders identically everywhere, unlike ↻ before it was replaced), 0.6×–2× in 0.2 steps, multiplying the point spacing. On its own row below the legend (flex-col on mobile, merges into the legend's row via sm:flex-row sm:ml-auto on desktop) — an earlier attempt put it inline in the legend's own flex-wrap row, where it became a 4th competing item and reintroduced the exact "wraps to two rows on mobile" bug the legend spacing/font-size had just been tuned to fix.

Filter modes (only shown for multi-season series, in a RatingChartSection wrapper in SeriesDetail.jsx — RatingChart itself stays a "dumb" renderer of whatever data it's given): a <select> for "Tutti gli episodi" (default, chronological) / "Per stagione" (adds a season <select>) / "Confronta stesso episodio tra stagioni" (adds an episode-number <select>, x-axis becomes season labels — S1, S2, ... — instead of episode labels). RatingChart gets a key built from the current mode/season/ episode selection, remounting it (and resetting zoom/hover) on every filter change instead of carrying state over from a completely different view.

Chart color tokens — --chart-blue/--chart-purple in src/index.css, intentionally separate from the semantic UI palette above: these need to read as identity (person A vs. person B), not app state, and were validated as a categorical pair (CVD separation + contrast against bg-surface, light and dark) with the dataviz skill's validator script. Don't repurpose them for anything else or swap the hex values without re-running that validation.

Calendario (month grid)

src/pages/Calendar.jsx — month-grid layout, mobile-first: 7 columns (Lun-Dom), each cell h-12 sm:h-14 rounded-lg border showing the date number plus one small dot (h-1.5 w-1.5 rounded-full) per scheduled/watched entry — bg-accent-solid for real watch history, bg-muted for future projected entries. Selected day: border-accent-solid bg-accent-soft. Today (unselected): border-accent-solid bg-surface. Tapping a cell selects it and shows its full entry list below in the same rounded-xl border border-border bg-surface px-3.5 py-2.5 row pattern used elsewhere; watched entries get an inline type="date" input to correct the recorded date. Future entries are derived live (never stored) from each series' watchDays + remaining unwatched count; past/today entries come from the real per-episode watched date — see src/lib/schedule.js. Navigation (← → buttons, mouse wheel, touch swipe — all three active at once) triggers a direction-aware slide+fade replay: the grid is remounted via a React key tied to ${year}-${month} with class animate-slide-next (from the right) or animate-slide-prev (from the left) depending on which way the month changed — keyframes in src/index.css. Wheel navigation is throttled with a short 150ms cooldown (a useRef lock, not React state) just enough to stop one trackpad gesture firing several months at once, without feeling laggy on deliberate fast scrolling.
